# Cybersecurity Intrusion Detection — Server Log Analysis Demo

A hands-on student lab that teaches how to detect real-world cyberattacks by
analysing three standard log formats: Apache web server access logs, Linux SSH
authentication logs, and iptables firewall drop logs.

---

## What This Demo Teaches

| Skill Area | What You Learn |
|---|---|
| Log formats | Apache Combined Log, Linux syslog, iptables/netfilter kernel format |
| Attack recognition | Reconnaissance scans, SQL injection, path traversal, XSS, brute force, HTTP flood, port scan |
| Detection algorithms | Regex-based pattern matching, sliding-window rate analysis, cardinality counting |
| Python data engineering | Parsing unstructured text into pandas DataFrames |
| Visualisation | Building security dashboards with matplotlib |

## Quickstart

```bash
# 1. Install dependencies
pip install -r requirements.txt

# 2. Run the full analysis (all three logs)
python intrusion_detection.py

# 3. Analyse a single log file
python intrusion_detection.py --log data/apache_access.log
python intrusion_detection.py --log data/ssh_auth.log
python intrusion_detection.py --log data/firewall.log

# 4. Regenerate logs with a different random seed
python generate_logs.py --seed 99

# 5. Generate a longer dataset (5 days)
python generate_logs.py --seed 7 --days 5
```

The script exits with **code 1** if any CRITICAL-severity finding is detected,
making it easy to integrate into a CI/CD pipeline or alerting workflow.

## Detection Algorithms

### 1. SSH Brute Force (`detect_ssh_brute_force`)
**Technique**: Sliding-window counter.

For each source IP, the algorithm sorts failed-password events by timestamp
then uses a two-pointer sweep. For every event `i`, it counts how many
subsequent events fall within a configurable window (default: 10 minutes).
If the count reaches the threshold (default: 10), the IP is flagged.

```
Triggers when: ≥ 10 failed SSH logins within any 10-minute window
Severity: CRITICAL
```

### 2. SQL Injection (`detect_sqli`)
**Technique**: Regex on URI field.

Patterns searched (case-insensitive):
```
UNION SELECT, OR 1=1, AND 1=1/2, DROP TABLE, INSERT INTO,
exec(, xp_, SLEEP(, EXTRACTVALUE(, information_schema,
URL-encoded variants (%20UNION%20, %27+OR+%271),
comment truncation (--, #, /*)
```

### 3. Path Traversal (`detect_path_traversal`)
**Technique**: Regex on URI field.

Patterns:
```
../ and ..\ (Unix/Windows), %2e%2e (URL-encoded ..)
..%2f, %2e%2e%2f, %2e%2e%5c, ....// (double-dot obfuscation)
```

### 4. Cross-Site Scripting (`detect_xss`)
**Technique**: Regex on URI field.

Patterns:
```
<script, javascript:, onerror=, onload=, onmouseover=,
alert(, %3cscript (URL-encoded), <svg, <img src
```

### 5. Malicious Scanner UA (`detect_bad_agents`)
**Technique**: Regex on User-Agent field.

Tool signatures detected:
```
sqlmap, nikto, nessus, masscan, nmap, zgrab, dirbuster, gobuster,
nuclei, havij, acunetix, metasploit, w3af, burpsuite, openvas,
Go-http-client (common in automated flood tools)
```

### 6. HTTP Rate Anomaly / Flood (`detect_rate_anomaly`)
**Technique**: Per-IP, per-minute bucketing.

Requests are grouped by `(IP, floor(timestamp, 1 minute))`.
Any bucket exceeding `rpm_threshold` (default: 20 req/min) triggers an alert.

```
Triggers when: ≥ 20 requests from one IP in any single clock-minute
Severity: HIGH
```

### 7. Port Scan (`detect_port_scan`)
**Technique**: Unique destination-port cardinality per source IP.

Counts how many distinct destination ports each source IP has hit in the
firewall drop logs. Scanning tools (nmap, masscan, zmap) generate characteristic
patterns of many unique ports in rapid succession.

```
Triggers when: ≥ 15 unique destination ports from one IP
Severity: HIGH
```

## How to Regenerate Logs

The static log files in `data/` are committed for convenience, but you can
regenerate them at any time:

```bash
# Default (seed=42, 3 days starting 2026-05-19)
python generate_logs.py

# Different seed — same attack structure, different timestamps/values
python generate_logs.py --seed 99

# 5-day dataset
python generate_logs.py --seed 7 --days 5
```

Output summary:
```
[+] apache_access.log :   213 lines  ->  data/apache_access.log
[+] ssh_auth.log      :   185 lines  ->  data/ssh_auth.log
[+] firewall.log      :    97 lines  ->  data/firewall.log

[✓] Done. 495 total log lines generated (seed=42, days=3).
```

## Learning Exercises

**Exercise 1 — Lower the brute-force threshold**
Edit `intrusion_detection.py` and change `detect_ssh_brute_force(threshold=10)`
to `threshold=5`. Re-run the script. Does it still catch the same attacker?
What new false positives (if any) does a lower threshold introduce?

**Exercise 2 — Add a new SQLi payload**
Open `data/apache_access.log` and manually add a line with the payload
`/api/users?id=1' WAITFOR DELAY '0:0:5'--` (MSSQL time-based blind injection).
Run the detector. Does it catch it? If not, update the `detect_sqli` regex
in `intrusion_detection.py` to cover MSSQL syntax.

**Exercise 3 — Trace the post-compromise timeline**
Open `data/ssh_auth.log` and read the entries after the successful brute force
at `23:52:07`. List all the commands the attacker ran and explain what each one
does. Which command is the most dangerous and why?

**Exercise 4 — Port scan fingerprinting**
Count how many of the ports scanned by `198.51.100.55` (in `firewall.log`)
correspond to database services. Look up each port number and categorise all
35 ports by service type (web, database, remote access, messaging, etc.).

**Exercise 5 — Improve the dashboard**
Add a 7th plot panel (make the grid 2×4 or 3×3) showing the geographic
distribution of attacker IPs using a free IP-geolocation library such as
`ip2geotools`. Note which RFC 5737 ranges are documentation-only addresses
and what real-world IP ranges are used for actual attacks.

## Log Format Quick Reference

### Apache Combined Log Format
```
%h %l %u %t "%r" %>s %b "%{Referer}i" "%{User-Agent}i"
10.0.1.50 - - [19/May/2026:08:03:12 +0000] "GET / HTTP/1.1" 200 4523 "-" "Mozilla/5.0 ..."
```

### Linux Syslog (SSH)
```
Month DD HH:MM:SS hostname service[pid]: message
May 19 08:05:22 webserver01 sshd[10101]: Accepted publickey for deploy from 10.0.1.50 ...
```

### iptables / netfilter Kernel Log
```
Month DD HH:MM:SS hostname kernel: [uptime] PREFIX IN=iface SRC=x DST=y ... DPT=n
May 20 13:45:02 webserver01 kernel: [94462.100001] IPTABLES-DROP IN=eth0 ... SRC=198.51.100.55 DPT=21
```
